# Stage 4a pilot — N=100, what it produced

**Date:** 2026-05-03
**Run:** `~/work/eunosia/artifacts/medqa-stage-4a-pilot-n100/`
**Purpose:** de-risking smoke before the 12-15h N=1273 stage-4a run.
**Verdict:** signal-AMBIGUOUS tilting negative; AUC headlines unreliable
because Condition C dropped 18 % of trajectories. The 18 % failure rate is
the load-bearing diagnostic, not the AUC numbers.

## What ran

```
python experiments/medqa_generalization/scripts/03_pipeline_validation_ollama.py \
    --n-questions 100 \
    --embedder-backend sentence-transformers \
    --embedder-model intfloat/multilingual-e5-large \
```

qwen2.5:7b-instruct via Ollama on M1 Pro.
Defaults otherwise: `temperature=0.0`, default decomposer config,

Wall-clock: ~100 minutes for 100 questions × 3 conditions
(~1 q/min including initial CoT for A/B/C plus per-step distribution
batch for C). Extrapolates to ~21 h for the full N=1273; about 50 %
slower than the original 12-15h estimate.

## Headline numbers

| Cond | AUC | n_pos / n_neg | n_failures |
|------|-----|---------------|------------|
| A (CoT only)         | 0.500 | 38 / 62 | 0  |
| B (CoT + confidence) | 0.580 | 38 / 62 | 0  |
| **C (CoT + signature composite)** | **0.447** | 38 / 62 | **18 / 100 (18 %)** |

A's 0.500 is correct by construction (constant signal). Recovery
produced 429 nodes / 347 edges / 429 visits — sparse but present.

Top-20 high-signature trajectories: 13 / 20 are
`high_score_correct_outcome=False` (high signature on correctly-answered
questions). At face value this looks like the signature is anti-tracking
correctness; with N=82 effective for C, it is also explainable as noise.

## Why "signal-AMBIGUOUS" not "signal-negative"

Pre-committed criteria (`stage_4a_pre_run_analysis_plan.md`) interpret
this run mechanically:

- Signal-positive needs C ≥ 0.55 with CI lower bound ≥ 0.50. Fails.
- Signal-negative trips on C < 0.55 OR (C and B AUCs indistinguishable
  with overlapping CIs). Trips both.

But the criteria assumed comparable conditions: same N, same drop-out,
no condition-specific failure mode. The pilot violated this assumption
substantially:

- C's effective N is 82, vs A and B at 100.
- The 18 dropped questions may be systematically the harder cases
  (qwen2.5:7b producing unparseable distributions on questions where it's
  most uncertain), creating an unmeasured selection bias in C's
  remaining sample.
- N=82 with bootstrap CI is wide enough that 0.447 honestly straddles
  0.40-0.55.

The pre-committed criteria don't apply mechanically when the setup
isn't comparable. Invoking the signal-negative branch literally
("C is below chance, abandon the framework") would over-claim from
this evidence.

The honest reading: the framework's empirical headline is not
evaluable until the 18 % failure rate is reduced.

## Bug surfaced and fixed

`_checkpoint_trajectories` and the recovery site in
`03_pipeline_validation_ollama.py` were not filtering `success=False`
results. The `ConditionResult` contract (`result.py:61`) says the runner
must filter before persistence and signature scoring, but the script
wasn't enforcing it. Condition C's `_build_failure_result` constructs
a minimal trajectory with `embedding=None`, which violates recovery's
uniform-embedding invariant.

Result: pipeline ran all 100 questions through Ollama (LLM work
completed), then crashed at `recover_assembly_graph` on the first
failed Condition C trajectory (`medqa-test-5`).

Two-line fix shipped (commit `e0f4d11`). `--resume` picked up cleanly
and produced the numbers above. **The same bug would have killed the
N=1273 run at hour ~12 of 12-15. Pilot served its purpose.**

## The 18 failures, characterised

100 % of failures (18 / 18) carry `failure_reason =
"hypothesis_distribution_batch_failed"`. Single failure mode.
All 18 carry `n_llm_calls=3, repair_attempts=1` — initial CoT
succeeded, decomposer succeeded (`decomposer_warnings: []`), the
post-CoT hypothesis-distribution batch failed and surgical repair
also failed.

Direct re-test on 5 failed cases (re-running just the
hypothesis-distribution prompts against Ollama):

| qid             | timestep | sum   | parses | sum-valid |
|-----------------|----------|-------|--------|-----------|
| medqa-test-5    | prior    | 1.10  | yes    | NO        |
| medqa-test-5    | step3    | 1.00  | yes    | yes       |
| medqa-test-6    | prior    | 1.00  | yes    | yes       |
| medqa-test-6    | step3    | 1.00  | yes    | yes       |
| medqa-test-8    | prior    | **2.00** | yes | NO        |
| medqa-test-8    | step3    | 1.00  | yes    | yes       |
| medqa-test-12   | prior    | 1.00  | yes    | yes       |
| medqa-test-17   | prior    | 1.00  | yes    | yes       |

Two patterns visible:

1. **Mild rounding** (medqa-test-5):
   `{"A":0.05, "B":0.1, "C":0.2, "D":0.75}` → sum 1.10.
   Model intended a distribution; arithmetic was sloppy.

2. **Semantic confusion** (medqa-test-8):
   `{"A":0.9, "B":0.85, "C":0.05, ...}` → sum ≈ 2.0.
   Model treated entries as independent likelihoods (P(A | data)
   evaluated separately for each option), not as a normalized
   distribution. Relative ranking still meaningful but the
   normalization invariant is broken.

`_parse_and_validate` (`llm_local.py:331`) requires
`math.isclose(total, 1.0, abs_tol=1e-6)`. With `temperature=0.0`,
retries are deterministic — same prompt yields same broken output.
Surgical repair re-issues the same prompt at the same temperature
and fails for the same reason. Per-question failure cascades from a
single bad prompt out of N+1 timesteps.

Cases 6/12/17 didn't reproduce on the live re-test for the prior
prompt; they presumably failed on a different timestep in the
original run (the script tested only prior + step3 here, not all
N+1 timesteps).

## Hypothesis

The dominant failure mechanism is qwen2.5:7b producing distributions
that don't sum to 1.0, combined with strict `1e-6` validation and
deterministic-retry. This is a tractable bug, not a fundamental
framework or model-capability problem.

Two patterns argue for different fixes:

- The 1.10-style rounding error is a near-miss; normalising to 1.0
  preserves the model's intent and recovers the data.
- The 2.0-style independent-likelihood pattern reflects a genuine
  prompt-comprehension gap; the model isn't producing what we asked
  for. Normalising still preserves ranking and is probably acceptable
  for signature scoring (the signature uses entropy and KL, both
  invariant to scale-normalization), but it's a weaker signal than
  a model that natively returned a distribution.

## Other observations worth keeping

- A's 0.500 AUC: framework's flat-line baseline holds, as expected.
  (Implicit confirmation that the comparison harness is correctly
  constructed.)
- B's 0.580 AUC: confidence-based deferral has weak but above-chance
  signal at this N. Worth noting but not headline-relevant; the
  comparison is C vs B, and C is currently unevaluable.
- Recovery: 429 nodes / 347 edges / 429 visits on 82 successful
  trajectories. Avg out-degree ~0.81 — sparse. Most edges likely
  freq=1; the embedding-bin precision sweep (ADR-0002) is the
  obvious follow-up for graph density.
- Wall-clock: 1 q/min on M1 Pro suggests the original 12-15h budget
  for N=1273 was optimistic; realistic is closer to 21h. Not a
  blocker for stage 4b H100 budget but worth noting for the
  M1 Pro-only path.

## What to NOT conclude

- ❌ "The framework doesn't work on MedQA" — N=82 with selection
  bias, you can't tell yet.
- ❌ "qwen2.5:7b is too small for this task" — initial CoT and
  decomposer worked on 100/100. The failure is in one specific
  output-format invariant.
- ❌ "Move on to a larger model" — the bug fix is much cheaper
  than commissioning H100 time.

## Next experiment (proposal — see follow-up doc)

Repeat N=100 pilot with:

1. Validation tolerance relaxed to `abs_tol=0.1` AND auto-normalise
   when the sum is in `[0.5, 2.5]` and falls outside `1e-6` of 1.0.
   Reject (raise) only outside `[0.5, 2.5]` — that catches
   semantically broken outputs (e.g., negative values, sum=0,
   sum=10) while accepting the two patterns observed above.
2. (Optional) Repair path bumps temperature to ~0.3 instead of
   re-issuing at 0.0. Decoupled from the normalization fix; do this
   only if normalize-and-accept doesn't drive failure rate to
   acceptable levels (<2 %).

Acceptance criterion for re-run: Condition C failure rate < 2 %
(comparable across conditions), then read the AUCs at face value
against the original pre-committed criteria.

If failure rate drops and C AUC is still below 0.55, the framework
genuinely lacks signal on this benchmark configuration and we move
to the diagnostic-cascade plan (per-component AUCs, embedding-bin
sweep, prompt-variant sweep). If failure rate drops and C AUC
crosses the 0.55 threshold, signal-positive interpretation applies
and stage 4b H100 scoping becomes appropriate.
